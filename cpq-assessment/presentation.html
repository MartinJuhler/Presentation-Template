The Consultant Era

18 Months of Effort.
Nothing Delivered.

The Infor CPQ project was run by an external consultant (Liselotte) for over a year. Here is what happened — and where it was left.

Project Start
Initial Workshop — No Documentation
Single workshop with Infor. No documentation produced. Setup built on a myriad of interdependent matrices that reference each other — but updating one does not propagate to the others. The visual interface is so convoluted it's extremely hard to navigate or trace dependencies. Participants were highly confused by the setup — it's not a user-friendly tool, and effectively requires coding knowledge to operate.
~4-6 Months In
Products Built — None Fully Tested
Multiple product configurations were built (with Vincente from Infor), tracked in a spreadsheet. However, none were ever fully tested, and many were noted as incorrect. Pricing was not even considered — only added later as a costly custom integration. This shows that the use case was never properly thought through before building — we just went at it.
Mid-Project
Consultant Reassigned — CPQ Abandoned
Liselotte switched to EPM and data migration (agreed with AKVA's project lead). CPQ maintenance was not prioritized. She stated clearly: "CPQ was not mature for use."
Oct 2024
ERP Goes Live — CPQ Left Behind
ERP went live on production. CPQ matrices were never synced with production data. The data refresh between environments broke existing configurations.
Late 2025
Consultant on Parental Leave — No Knowledge Transfer
Liselotte unavailable until Spring 2026. No other person has full system knowledge. Only documentation is what exists on AKVA's Teams site. Her manager confirmed: a new developer would need to start with a gap analysis.
"CPQ was not mature for use — AKVA Group was aware of this. The solution will require changes in collaboration with Infor."
— Liselotte's manager, formal status update

Why It's Fundamentally Broken

🔄
No Automatic Data Sync
Prices are fetched dynamically via FnOCOLL, but product data, item lists, and options are 100% static. These must be manually updated for tens of thousands of components. Any change in the ERP — renamed items, new products, removed parts — breaks CPQ silently.
📂
Scattered Manual Files
Configuration data is spread across manually maintained spreadsheets and matrices. Mixture of old and new product numbers. No single source of truth.
💥
No Internal Safety Logic
No logic prevents cascading failures when data changes. If an item is updated, renamed, or removed in ERP, CPQ breaks silently — producing wrong quotes or crashing entirely.
"The amount of parts we have is just so big that it's not a manual task. Without solving the update of the matrices as a foundation, I don't see this solution is workable for AKVA."
— CPQ Debugging Session, Nov 2025
"Often, they didn't come prepared; they could only solve bits of the puzzle as they only know a little part of the product, so new meetings with multiple people had to be scheduled to come to a full understanding."
— Internal project review

The Fundamental Problem

A Legacy Tool That Can't
Adapt to AKVA's Reality

Even if we invested heavily, Infor CPQ's rigid architecture makes it structurally incapable of handling AKVA's product complexity — now or in the future.

⚙️ Can't Handle Our Complexity
AKVA's products — floating feeding modules (flåte), water transport, nets — require complex, tailored calculations for correct quotations. Infor CPQ relies on rigid rules designed for standard manufacturing products. It cannot perform the computations needed without extreme cost and workarounds. Products like flåte would never work in this architecture.
📦 BOM Splitting Requires Additional Work
Infor CPQ currently sends the entire BOM under one header, which our ERP can't use for multi-site production. A side project was started to address this, but it was put on hold before we received a cost or time estimate. It would require additional investment on top of everything else — for a feature that should be fundamental.
👥 One Developer at a Time
Only one person can work in the project at a time. No branching, no merging — just push from server to server. Changes traced through manual comments only. No automated testing before going live. This doesn't scale.
🔮 Only Solves Standard Products
Infor's proposed new approach uses 2-3 large matrices. Items could be synced from the ERP into one — but the options matrix would still need manual maintenance. Critically, this approach would only work for standard products — complex products like flåte are simply not solvable within this architecture.
"Choosing Infor CPQ would almost certainly generate strong pushback from other branches, impacting the likelihood of running a successful project."
